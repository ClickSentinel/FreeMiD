### 1. Rust Code Quality

#### What is done well

- **Error handling** is consistent — `Result<T, String>` and a proper `IpcError` enum throughout; no silently-swallowed errors.
- **`unsafe` is minimal and justified** — only the Windows `CreateMutexW`/`CloseHandle` FFI calls, correctly wrapped in a RAII `SingleInstanceGuard` with a `Drop` impl (`main.rs:119-133`).
- **Binary update is carefully ordered**: SHA-256 verify → apply; the binary never touches disk until the checksum passes.
- **Unix update uses `rename(2)`** (truly atomic); Windows correctly uses a staged file + helper binary.
- **Release profile is well-tuned**: `opt-level = "z"`, `lto = true`, `codegen-units = 1`, `strip = true`, `panic = "abort"`.

## Part 2: Auto-Update Process — Deep Technical Investigation

---

### Happy Path (end-to-end walkthrough)

#### Step 1 — User clicks "Update" in the popup

`popup/index.ts:243` — the `btnUpdate` click handler fires. Two early-exit checks occur first:

- If `hostSelfUpdateSupported === false` (explicit legacy host signal) → opens the install guide instead.
- If on Windows and the error matches "unsupported platform" or "manual bootstrap required" → opens the install guide.

If neither condition applies, `chrome.runtime.sendMessage({ type: 'RUN_HOST_UPDATE' })` is sent.

#### Step 2 — Background receives `RUN_HOST_UPDATE` (`background/index.ts:707`)

1. Ensures a native port exists; fails fast if connection cannot be established.
2. Checks `hostSelfUpdateSupported !== true` — fails with `manualInstallRequiredError()` rather than sending the command. Guards against legacy hosts getting stuck in `requested` forever.
3. Calls `isHostSelfUpdateSupported(hostVersion, ...)` — a version floor check against `MIN_SELF_UPDATE_HOST_VERSION = '0.4.0'`. Fails identically if below minimum.
4. Sets `updateStatus = { status: 'requested' }` and broadcasts — the popup spinner starts.
5. Sends `{ type: 'UPDATE', latestUrl?, releasesBaseUrl? }` to the native host. URL overrides come from `VITE_UPDATE_LATEST_URL` / `VITE_UPDATE_RELEASES_BASE` baked in at extension build time.
6. Arms `updateRequestTimeoutTimer` — if the host doesn't reply within `UPDATE_REQUEST_TIMEOUT_MS` (8 s on Linux/macOS, 12 s on Windows), status transitions to `failed`.

#### Step 3 — Native host receives `UPDATE` (`main.rs:316-329`)

Extracts optional URL overrides, constructs `UpdateSourceOverrides`, calls `update::run_update(overrides, |v| write_message(&v))`.

`run_update` (`update.rs:68`) checks the `UPDATE_IN_PROGRESS` atomic bool via `swap(true)` — if already true, immediately sends `failed "Update already in progress"` and returns. Otherwise spawns a background thread calling `do_update()`.

#### Step 4 — Background thread `do_update()` (`update.rs:94`)

1. Sends `UPDATE_STATUS { status: "checking" }` → background clears the request timeout. State: `requested → checking`.
2. Fetches `https://api.github.com/repos/ClickSentinel/FreeMiD/releases/latest`.
3. Parses `tag_name`, strips leading `v`, enforces strict semver (`is_strict_semver` requires exactly 3 dot-separated numeric components).
4. Calls `is_newer(latest, current)` comparing as `[u64; 3]` tuples. If not newer → sends `up_to_date` and exits.
5. Sends `UPDATE_STATUS { status: "downloading", version }`. State: `checking → downloading`.
6. Downloads binary via `download_bytes()`. Size-guarded at 100 MB via `Content-Length` pre-check and hard `.take(MAX_DOWNLOAD_BYTES + 1)`.
7. Downloads `checksums.sha256` from the same release tag directory via `download_string()`.
8. Calls `verify_sha256()` — parses the checksum file line by line, matches artifact name (strips `./` and `*` prefixes), compares expected hex to `sha2::Sha256` of the downloaded bytes.
9. Calls `apply_update()` (see platform differences below).

#### Step 5 — Extension receives `UPDATE_STATUS success` (`background/index.ts:408`)

1. Immediately transitions `updateStatus` to `{ status: 'reconnecting', version: targetVersion }`.
2. Calls `startApplyVerification(targetVersion)` — records deadline and begins polling every 1 s.
3. Sets `autoReconnectScheduled = true`.
4. Schedules `reconnectNativeHost()` after `POST_UPDATE_RECONNECT_DELAY_MS` (150 ms on Linux/macOS, 5 000 ms on Windows). State: `success → reconnecting`.

#### Step 6 — Reconnect

Chrome kills the old host process when the port disconnects. On the next `connectNativeHost()`, Chrome spawns whatever binary now sits at the registered path — the updated binary on Linux/macOS (renamed in-place) or the copied binary on Windows (after `freemid-apply` finishes).

#### Step 7 — Version verification

Every second, the `applyVerifyTimer` sends a PING and calls `maybeFinalizeAppliedVersion()` — checks `compareVersions(hostVersion, applyVerifyTargetVersion) >= 0`. On success: clears all verification state, sets `updateStatus = null`, broadcasts. The popup hides the update button.

If the deadline expires before the version advances: sets `updateStatus = { status: 'failed', error: 'Host version is still X after update' }`.

### Platform Differences

#### Linux / macOS

- Apply is an atomic single-call `rename(2)` in-process (`update.rs:350-392`).
- The running process continues from the old inode; the filesystem entry now points to the new binary.
- The process does **not** exit after apply — it sends `success` and waits for Chrome to disconnect it.
- Reconnect delay: 150 ms. Apply verification timeout: 30 s.

#### Windows — why two-process

Windows holds an exclusive file lock on a running `.exe`. A `rename()` or `copy /Y` to overwrite a running executable fails with `ERROR_SHARING_VIOLATION`. The original process must exit first to release the lock.

**Two-process sequence:**

1. `freemid.exe` writes staged file (`freemid.exe.staged-<pid>.exe`), spawns `freemid-apply.exe`, sends `success`, sleeps 120 ms, then `process::exit(0)` (`update.rs:181-186`).
2. `freemid-apply.exe` polls `fs::copy(staged → target)` every 100 ms for up to 60 seconds waiting for the old process's file lock to clear.

**Fallback chain on Windows:**

1. Stable `freemid-apply.exe` helper (preferred — pre-installed, stable binary path).
2. If helper missing or returns OS error 740 (elevation required): `cmd.exe /C copy /Y` loop for up to 120 s.

**Single-instance guard:** `CreateMutexW("Local\FreeMiD.NativeHost")` with 3 retries at 750 ms each — prevents two host processes racing on reconnect when the old one hasn't fully exited (`main.rs:66-72`).

Reconnect delay: 5 000 ms. Apply verification timeout: 130 s.

### State Machine

```
null
 └─► requested        (background sets on RUN_HOST_UPDATE, before sending UPDATE)
      └─► checking     (host sends UPDATE_STATUS checking)
           └─► downloading  (host sends UPDATE_STATUS downloading)
                ├─► up_to_date  (host sends UPDATE_STATUS up_to_date → status clears to null)
                ├─► failed      (any error from host: network, checksum, apply failure, reentrancy)
                └─► success     (host sends UPDATE_STATUS success)
                     └─► reconnecting  (background sets immediately on success receipt)
                          ├─► null     (version verified ≥ target → status clears to null)
                          └─► failed   (APPLY_VERIFY_TIMEOUT_MS expired; host version still old)
```

**Additional transitions not driven by host messages:**

- `requested → failed`: `UPDATE_REQUEST_TIMEOUT_MS` fires before any `UPDATE_STATUS` arrives.
- `requested → failed`: `STATUS` arrives while status is `requested` but `hostSelfUpdateSupported !== true`.
- Any in-flight state (`requested`, `checking`, `downloading`, `reconnecting`) → `reconnecting`: host disconnects mid-update.
- `reconnecting → null`: `maybeFinalizeAppliedVersion()` succeeds (from `applyVerifyTimer` tick or an incoming STATUS).

### Security Properties

#### What IS verified

| Check | Where |
|-------|-------|
| SHA-256 checksum of the downloaded binary against `checksums.sha256` | `update.rs:312-341` |
| HTTPS enforced for all non-localhost URLs | `update.rs:232-249` |
| Strict semver format on the release tag before any download | `update.rs:132-136` |
| Version monotonicity — no downgrades from the default GitHub source | `update.rs:138-141` |
| Windows `freemid-apply.exe` validates exact filenames and same-directory constraint | `update.rs:551-583`, `updater_main.rs:64` |
| Download size capped at 100 MB | `update.rs:269` |

#### What is NOT verified

- **No code signing.** If a GitHub release is compromised (e.g., stolen token), an attacker can replace both the binary and the checksum file simultaneously — SHA-256 verification still passes.
- **`checksums.sha256` is not GPG/Sigstore signed.** Security depends entirely on the HTTPS transport to `github.com`.
- **Source override URLs are not restricted to `github.com`.** A dev build with `VITE_UPDATE_RELEASES_BASE` pointing elsewhere can serve any version newer than current.

### Failure Modes

| Scenario | What happens |
|----------|-------------|
| Network drops mid-download | `ureq` returns an error → propagates to `failed` status; staged binary (in memory only) is never written; `UPDATE_IN_PROGRESS` resets to false |
| Checksum mismatch | `verify_sha256` returns `Err` → `failed` status; `apply_update` is never called |
| `freemid-apply.exe` crashes silently (Windows) | Staged file remains on disk; old binary unchanged; Chrome reconnects, spawns old binary; `applyVerifyTimer` eventually times out at 130 s → `failed`; staged file is **not cleaned up** (orphan leak) |
| `cmd.exe` fallback loop times out | Extension already received `success` and is in `reconnecting`; same 130 s timeout outcome as above |
| Version doesn't advance after reconnect | `applyVerifyTimer` polls every 1 s; after `APPLY_VERIFY_TIMEOUT_MS` sets `failed "Host version is still X after update"` |
| Host doesn't respond to `UPDATE` | `UPDATE_REQUEST_TIMEOUT_MS` fires → `failed "Host did not acknowledge update command"` |
| Host disconnects during `checking`/`downloading` | `onDisconnect` sets `reconnecting`, starts `DISCONNECT_RECONNECT_DELAY_MS` timer to reconnect; if binary wasn't applied, version check eventually times out |
| Service worker suspended between `success` and reconnect timer | `autoReconnectScheduled` flag lost; reconnect never happens; user stuck in `reconnecting` until 130 s timeout |
